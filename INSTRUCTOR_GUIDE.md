# Инструкция для проверяющего (преподавателя)

## Что было реализовано

В соответствии с сообщениями из чата была реализована возможность:
1. **Доступа к приложению по IP адресу** (не только localhost)
2. **Работы GitHub Pages с локально запущенным бэкендом**

## Основная идея (из чата)

> "GH вроже должен по https общаться с твоим локальным фронтом, а фронт проксироватт запросы на бэк"
> 
> "Так он же писал, что подключаемся к ip тачки в локальной сети (==wifi). Типо: ip ноута 192.168.1.187, а телефона - 192.168.1.56. Вот и нужно их подружить"
>
> "Главное сервак запускать на 0.0.0.0:твой порт"

## Как проверить работу

### Вариант 1: Проверка локального доступа по IP

1. **Запустите проверку конфигурации:**
   ```bash
   ./check_config.sh
   ```
   Это покажет IP адрес и состояние конфигурации.

2. **Запустите бэкенд на 0.0.0.0:**
   ```bash
   ./run_backend.sh
   ```
   Скрипт покажет IP адрес машины. Например: `192.168.1.240`

3. **Запустите фронтенд на 0.0.0.0:**
   ```bash
   ./run_frontend.sh
   ```
   В другом терминале. Также покажет IP адрес.

4. **Проверьте доступ:**
   - С локальной машины: `http://localhost:3000`
   - С телефона в той же WiFi: `http://192.168.1.240:3000` (замените на ваш IP)

### Вариант 2: Проверка GitHub Pages с локальным бэкендом

1. **Настройте IP для GitHub Pages:**
   ```bash
   ./setup_ip_for_ghpages.sh
   ```
   Скрипт автоматически определит IP и обновит конфигурацию.

2. **Запустите бэкенд:**
   ```bash
   ./run_backend.sh
   ```
   Бэкенд должен работать на `0.0.0.0:8000`.

3. **Соберите и задеплойте фронтенд:**
   ```bash
   cd frontend
   npm run build
   npm run deploy
   ```
   Или используйте автоматический скрипт:
   ```bash
   ./deploy_to_ghpages.sh
   ```

4. **Откройте GitHub Pages:**
   ```
   https://YurchenkoK.github.io/vasoactive_drug_speed_estimatior_frontend/
   ```
   
5. **Проверьте работу:**
   - Откройте сайт на **любом устройстве в той же WiFi сети**
   - Приложение должно работать и делать запросы к локальному бэкенду
   - Откройте DevTools → Network и убедитесь, что запросы идут на `http://192.168.1.X:8000/api/...`

## Ключевые изменения в коде

### 1. Frontend: vite.config.ts
```typescript
export default defineConfig(({ mode }) => {
  return {
    server: { 
      host: '0.0.0.0',  // ← Доступ по IP
      proxy: {
        "/api": {
          target: `http://0.0.0.0:8000`,  // ← Прокси на бэкенд
        },
      },
    },
    // ...
  };
});
```

### 2. Frontend: drugsApi.ts

// Development: пустая строка (прокси)

const API_BASE_URL = isTauri

## Архитектура решения

### Development (локальная разработка)
```
Browser → http://192.168.1.X:3000 → Vite (прокси) → http://0.0.0.0:8000
```
- API_BASE_URL = '' (пустая строка)
- Все запросы `/api/*` проксируются через Vite
- Нет проблем с CORS (same origin)

### Production (GitHub Pages)
```
Browser → https://github.io/... (статика)
       ↓
       fetch('http://192.168.1.X:8000/api/...') → Django
```
- API_BASE_URL = 'http://192.168.1.X:8000'
- Прямые запросы от браузера к локальному бэкенду
- CORS настроен в Django

## Важные моменты

1. **0.0.0.0 vs 127.0.0.1:**
   - `127.0.0.1` - только локальная машина
   - `0.0.0.0` - все сетевые интерфейсы (доступ по IP)

2. **GitHub Pages:**
   - Отдает только статику (HTML/JS/CSS)
   - JavaScript делает запросы из браузера
   - Браузер подключается к локальному бэкенду по IP

3. **Прокси в Vite:**
   - Работает только в development режиме
   - В production нужен прямой URL
